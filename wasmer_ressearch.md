# Wasmer Runtime Architecture: JS to Rust Glue

## Overview

The wasmer-js project bridges JavaScript to the Wasmer WebAssembly runtime through **wasm-bindgen**. The Rust code is compiled to WebAssembly using `wasm-pack`, and JavaScript can call into it directly.

```
┌───────────────────────────────────────────────────────────────────┐
│  JavaScript (Browser/Node.js)                                     │
│  Wasmer.fromFile(bytes) → Promise<Wasmer>                        │
└───────────────────────────────┬───────────────────────────────────┘
                                │ wasm-bindgen
                                ▼
┌───────────────────────────────────────────────────────────────────┐
│  wasmer-js (Rust → WASM)                                         │
│  #[wasm_bindgen] js_from_file() → BinaryPackage → Wasmer struct  │
└───────────────────────────────┬───────────────────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────┐
│  wasmer-wasix (Rust crate)                                       │
│  WasiRunner, WasiEnv, BinaryPackage, spawn_exec()                │
└───────────────────────────────┬───────────────────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────┐
│  wasmer (Rust crate)                                             │
│  Engine, Module, Store - actual WASM compilation & execution     │
└───────────────────────────────────────────────────────────────────┘
```

## 2. Rust wasm-bindgen Layer

**File: `src/wasmer.rs:87-100`**

```rust
#[wasm_bindgen(js_name = "fromFile")]
pub async fn js_from_file(
    binary: Uint8Array,
    runtime: Option<OptionalRuntime>,
) -> Result<Wasmer, Error> {
    let bytes = binary.to_vec();
    if bytes.starts_with(b"\0asm") {
        // Raw WASM file - wrap in synthetic package
        Wasmer::from_wasm(bytes, runtime)
    } else {
        // WEBC package file - load container
        Wasmer::from_file(bytes, runtime).await
    }
}
```

**Key annotations:**
- `#[wasm_bindgen(js_name = "fromFile")]` - Maps Rust function to `Wasmer.fromFile` in JS
- `Uint8Array` - Automatically converted from JS `Uint8Array` to Rust vector
- `async fn` - Returns a JavaScript `Promise`

## 3. Loading a WEBC Package

**File: `src/wasmer.rs:126-134`**

```rust
async fn from_file(binary: Vec<u8>, runtime: Option<OptionalRuntime>) -> Result<Self, Error> {
    let runtime = runtime.unwrap_or_default().resolve()?.into_inner();

    // Detect WEBC format version
    let version = webc::detect(&mut (&*binary))?;

    // Parse WEBC container (zstd-compressed TAR with manifest + atoms)
    let container = webc::Container::from_bytes_and_version(binary.into(), version)?;

    // Convert to BinaryPackage (resolves dependencies, extracts commands)
    let pkg = BinaryPackage::from_webc(&container, &*runtime).await?;

    Wasmer::from_package(pkg, runtime)
}
```

## 4. Creating the Wasmer Struct with Commands

**File: `src/wasmer.rs:136-161`**

```rust
fn from_package(pkg: BinaryPackage, runtime: Arc<Runtime>) -> Result<Self, Error> {
    let pkg = Arc::new(pkg);
    let commands = Commands::default();  // JavaScript Object

    // For each command in the package, create a Command wrapper
    for cmd in &pkg.commands {
        let name = JsString::from(cmd.name());
        let value = JsValue::from(Command {
            name: name.clone(),
            runtime: Arc::clone(&runtime),
            pkg: Arc::clone(&pkg),
        });
        // Expose to JS: wasmer.commands.commandName
        Reflect::set(&commands, &name, &value).map_err(Error::js)?;
    }

    let entrypoint = pkg.entrypoint_cmd.as_deref().map(|name| Command {
        name: name.into(),
        pkg: Arc::clone(&pkg),
        runtime,
    });

    Ok(Wasmer { entrypoint, commands, pkg: None })
}
```

## 5. Running a Command

**File: `src/wasmer.rs:199-267`**

When you call `command.run()` in JavaScript:

```rust
#[wasm_bindgen]
impl Command {
    pub async fn run(&self, options: Option<SpawnOptions>) -> Result<Instance, Error> {
        // 1. Create pipes for stdin/stdout/stderr
        let (stdin, stdin_tx) = options.stdin.then(input_pipe).unzip();
        let (stdout, stdout_rx) = output_stream();
        let (stderr, stderr_rx) = output_stream();

        // 2. Configure WasiRunner
        let mut runner = WasiRunner::new();
        runner
            .with_args(args)
            .with_envs(env.iter().cloned())
            .with_stdin(Box::new(stdin_tx.unwrap_or_else(|| Box::new(Pipe::channel().0))))
            .with_stdout(Box::new(stdout_rx))
            .with_stderr(Box::new(stderr_rx));

        // 3. Spawn on dedicated thread
        let (exit_tx, exit_rx) = tokio::sync::oneshot::channel();
        tasks.task_dedicated(Box::new(move || {
            let result = runner.run_command(
                &command_name,
                &pkg,
                RuntimeOrEngine::Runtime(runtime)
            );
            let _ = exit_tx.send(ExitCondition::from_result(result));
        }))?;

        // 4. Return Instance with streams
        Ok(Instance { stdin, stdout, stderr, exit: exit_rx.shared() })
    }
}
```

## 6. Into wasmer-wasix: WasiRunner

**File: `../wasmer/lib/wasix/src/runners/wasi.rs`**

```rust
impl WasiRunner {
    pub fn run_command(
        &mut self,
        command_name: &str,
        pkg: &BinaryPackage,
        runtime_or_engine: RuntimeOrEngine,
    ) -> Result<(), Error> {
        // 1. Build WasiEnv from configuration
        let builder = self.prepare_webc_env(command_name, &wasi, pkg, runtime_or_engine, None)?;
        let env = builder.build()?;

        // 2. Load the WASM module for this command
        let cmd = package_command_by_name(pkg, command_name)?;
        let module = runtime.load_command_module(cmd).await?;

        // 3. Spawn execution
        let task_handle = spawn_exec_module(module, env, &runtime)?;

        // 4. Wait for completion
        let exit_code = task_handle.wait_finished().await?;
    }
}
```

## 7. Into wasmer-wasix: Module Loading

**File: `../wasmer/lib/wasix/src/runtime/mod.rs`**

```rust
pub trait Runtime {
    fn load_command_module(&self, cmd: &BinaryPackageCommand)
        -> BoxFuture<Result<Module, SpawnError>>
    {
        // Get WASM bytes + hash from command
        let data = HashedModuleData::from_command(cmd);

        // Try cache first, compile if miss
        load_module(self.engine(), self.module_cache(), &data)
    }
}

pub async fn load_module(
    engine: &Engine,
    module_cache: &dyn ModuleCache,
    module: &HashedModuleData,
) -> Result<Module, SpawnError> {
    let hash = module.hash();

    // Check cache
    if let Ok(cached) = module_cache.load(hash, engine).await {
        return Ok(cached);
    }

    // Compile from WASM bytes
    let compiled = Module::new(engine, module.wasm())?;

    // Save to cache
    module_cache.save(hash, engine, &compiled).await.ok();

    Ok(compiled)
}
```

impl Module {
    pub fn new(engine: &impl AsEngineRef, binary: &[u8]) -> Result<Self, CompileError> {
        Self::validate(engine, binary)?;

## Summary

The architecture cleanly separates concerns:

1. **wasmer-js** - Browser-specific: Web Workers, Web Streams, HTTP fetching
2. **wasmer-wasix** - WASI/WASIX runtime abstraction, environment management
3. **wasmer** - Core WASM compilation and execution

The glue is primarily `#[wasm_bindgen]` annotations that automatically generate JavaScript bindings, combined with manual stream bridging (`src/streams.rs`) and a custom thread pool implementation (`src/tasks/`) for Web Worker-based parallelism.

### Rust Layer (`src/`)

pub(crate) enum GlobalScope {
    Window(Window),                    // Browser main thread
    Worker(WorkerGlobalScope),         // Web Worker context
    Other(js_sys::Object),             // Unknown (Node.js, Deno, etc.)
}

impl GlobalScope {
    pub fn current() -> Self {
        let global_scope = js_sys::global();
        match global_scope.dyn_into() {
            Ok(window) => GlobalScope::Window(window),
            Err(global_scope) => match global_scope.dyn_into() {
                Ok(worker_global_scope) => GlobalScope::Worker(worker_global_scope),
                Err(other) => GlobalScope::Other(other),  // Fallback for Node.js
            },
        }
    }

    pub fn sleep(&self, milliseconds: i32) -> Promise { ... }
    pub fn user_agent(&self) -> Option<String> { ... }
    pub fn hardware_concurrency(&self) -> Option<NonZeroUsize> { ... }

## WASM Execution: Native WebAssembly APIs

**Both browser and Node.js use the same mechanism: native `WebAssembly` APIs.**

The `wasmer` crate has a **JS backend** (`lib/api/src/backend/js/`) that wraps the native `WebAssembly` JavaScript APIs. There is **no Wasmer VM running in WASM** - it delegates entirely to the host's WebAssembly implementation.

### Compilation (Module Creation)

**File: `wasmer/lib/api/src/backend/js/entities/module.rs:70-83`**

```rust
pub(crate) unsafe fn from_binary_unchecked(
    _engine: &impl AsEngineRef,
    binary: &[u8],
) -> Result<Self, CompileError> {
    let js_bytes = unsafe { Uint8Array::view(binary) };

    // Uses native WebAssembly.Module constructor
    let module = WebAssembly::Module::new(&js_bytes.into()).map_err(...)?;

    Ok(unsafe { Self::from_js_module(module, binary) })
}
```

This calls the **native** `new WebAssembly.Module(bytes)` - the browser/Node.js compiles the WASM to machine code.

### What Actually Runs the WASM

| Environment | WASM Engine | How It Works |
|-------------|-------------|--------------|
| **Chrome** | V8 | V8's TurboFan JIT compiles WASM to x64/ARM machine code |
| **Firefox** | SpiderMonkey | IonMonkey/Cranelift compiles WASM to native code |
| **Safari** | JavaScriptCore | B3 JIT compiles WASM to native code |
| **Node.js** | V8 | Same as Chrome - V8 compiles WASM to native code |
| **Deno** | V8 | Same as Chrome/Node.js |

### The wasmer Crate's Role

The `wasmer` crate in JS mode is essentially a **thin wrapper** around native WebAssembly APIs:

```
┌─────────────────────────────────────────────────────────┐
│  wasmer-js (Rust → WASM)                                │
│  Provides: WasiRunner, BinaryPackage, Runtime trait     │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐
│  wasmer crate [js backend]                              │
│  Wraps: js_sys::WebAssembly::Module                     │
│         js_sys::WebAssembly::Instance                   │
│         js_sys::WebAssembly::Memory                     │
│         js_sys::WebAssembly::Table                      │
└─────────────────────┬───────────────────────────────────┘
                      │ calls via wasm-bindgen
                      ▼
┌─────────────────────────────────────────────────────────┐
│  Native WebAssembly APIs (globalThis.WebAssembly)       │
│  Browser: V8/SpiderMonkey/JSC                           │
│  Node.js: V8                                            │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐
│  Native Machine Code (x64/ARM)                          │
│  JIT-compiled by the JS engine                          │
└─────────────────────────────────────────────────────────┘
```

### Key Point: No Interpreter

There's **no WASM interpreter** running. The wasmer-js SDK:

1. **Doesn't compile WASM itself** - delegates to `WebAssembly.Module`
2. **Doesn't execute WASM itself** - delegates to `WebAssembly.Instance`
3. **Does provide WASI/WASIX** - implements system calls (filesystem, networking, etc.)

The actual WASM bytecode is compiled to native machine code by V8/SpiderMonkey/JSC, running at near-native speed.

### Contrast with Native wasmer

On native platforms (not in browser), the `wasmer` crate has different backends:

| Backend | Compiler | Description |
|---------|----------|-------------|
| `sys` + Cranelift | Cranelift | Fast compile, good runtime |
| `sys` + LLVM | LLVM | Slow compile, best runtime |
| `sys` + Singlepass | Singlepass | Fastest compile, baseline perf |
| **`js`** | **Browser's JIT** | **Delegates to native WebAssembly** |

The `js` backend is used when `wasmer` is compiled to `wasm32-unknown-unknown` target.

### Filesystem Layering

```
┌─────────────────────────────────────────────────┐
│  WASI Process View                              │
│  /                                              │
│  ├── app/        ← Mounted from host           │
│  ├── tmp/        ← TmpFileSystem (writable)    │
│  ├── lib/        ← Package contents (read-only)│
│  └── ...                                        │
└─────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────┐
│  WasiFsRoot (base layer)                        │
│  TmpFileSystem or custom backing               │
└───────────────────┬─────────────────────────────┘
                    │
      ┌─────────────┼─────────────┐
      ▼             ▼             ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│  Mount 1 │  │  Mount 2 │  │  Union   │
│  /app    │  │  /data   │  │ Package  │
│ (host fs)│  │ (tmpfs)  │  │ contents │
└──────────┘  └──────────┘  └──────────┘
```

**Key operations:**

1. **Mount** - Attach a filesystem at a specific path:
   ```rust
   fs.mount("/app".into(), &host_dir, "/".into())
   ```

2. **Union** - Merge two filesystems (for package contents):
   ```rust
   fs.union(other);  // Merges package contents into sandbox
   ```

3. **Overlay** - Layer filesystems with read-through:
   - Reads check primary (writable) first, then fall back to secondaries
   - Writes go to primary layer only

### Instance-Specific vs Shared Filesystems in wasmer-js

Each spawned command (`Instance`) in wasmer-js gets its own `TmpFileSystem`:

```
┌─────────────────────────────────────────────────────┐
│  Instance 1 (e.g., bash -c "echo hello")            │
│  ├── TmpFileSystem (instance-specific, ephemeral)   │
│  └── /data ← Directory mount (SHARED)               │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  Instance 2 (e.g., ls /data)                        │
│  ├── TmpFileSystem (instance-specific, ephemeral)   │
│  └── /data ← Directory mount (SHARED)               │
└─────────────────────────────────────────────────────┘
```

**Implications:**
- Files written via `instance.vfsWriteFile('/foo')` are NOT visible to other Instances
- Files written to mounted Directory (`/data/*`) ARE visible to all Instances
- To test VFS operations with shell commands, use a single Instance with a shell script

### Persistence Options

| Method | How It Works |
|--------|--------------|
| **Host mounts** | Mount host directories via `runner.with_mount()` - writes go directly to host FS |
| **VFS API** | Read files via `instance.vfsReadFile()` before the instance is dropped |
| **Journal system** | Wasmer tracks syscalls for replay/restore (native only) |
| **Custom backing FS** | Provide a custom `FileSystem` impl that persists (requires Rust changes) |

### Why No Persistent Root FS in wasmer-js?

1. **`host_fs` requires native access** - Uses `std::fs` and `tokio::fs`, unavailable in WASM
2. **No browser filesystem API** - No standard persistent "root filesystem" in browsers
3. **Mounts are the design** - Expected pattern is to mount specific directories for persistence

## Host Execution (host_exec) IPC Architecture

### Overview

The `host_exec` feature allows WASM programs to execute commands on the host system (Node.js or browser). This is implemented via a message-passing IPC system between Web Workers (where WASM runs) and the main thread (where the JavaScript host exec handler runs).

### Message Flow

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  Worker Thread (WASM Execution)                                              │
│                                                                              │
│  1. WASM calls host_exec_start(command, args, env, cwd)                     │
│  2. Rust serializes request to JSON                                          │
│  3. Sends HostExecStart message via postMessage                             │
│  4. Returns session_id immediately (non-blocking)                           │
│                                                                              │
│  5. WASM calls host_exec_read(session_id) to get result                     │
│  6. Sends HostExecRead message via postMessage                              │
│  7. Awaits response... ⚠️ BLOCKS HERE                                       │
│                                                                              │
│  ⚠️ PROBLEM: While awaiting, the worker's event loop cannot process         │
│     incoming postMessage responses!                                          │
└─────────────────────────────────────────────────────────────────────────────┘
                              │
                              │ postMessage
                              ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  Main Thread (Scheduler)                                                     │
│                                                                              │
│  1. Receives HostExecStart message                                           │
│  2. Calls JS host_exec_handler(ctx) → returns Promise                       │
│  3. Spawns async task to await Promise                                       │
│  4. Sends HostExecStartResponse with session_id                             │
│  5. When Promise resolves, stores result in HOST_EXEC_RESULTS               │
│                                                                              │
│  6. Receives HostExecRead message                                            │
│  7. Looks up result in HOST_EXEC_RESULTS                                     │
│  8. Sends HostExecReadResponse with exit code                               │
│                                                                              │
│  ⚠️ PROBLEM: Steps 6-8 never happen because worker can't send               │
│     HostExecRead while blocked in step 7 on worker side!                    │
└─────────────────────────────────────────────────────────────────────────────┘
```

### The Fundamental Limitation

**Web Workers use an event-driven model with postMessage.** The worker's JavaScript event loop processes messages from its message queue. When WASM is executing (including when Rust async code is awaiting), the worker's event loop is blocked and cannot process incoming messages.

This creates a deadlock:
1. WASM calls `host_exec_read()` which awaits a response
2. The await blocks the worker's event loop
3. The main thread sends the response via `postMessage`
4. The response sits in the worker's message queue
5. The message cannot be processed because the event loop is blocked
6. Timeout occurs

**Evidence from test output:**
```
[host_exec] command=node args=["-e","console.log('hello from node')"]
hello from node                                          ← Command executed!
[host_exec] process exited with code: 0                  ← Promise resolved!
[scheduler] Promise resolved with exit_code=0 for session 0
[scheduler] pending read for session 0: None             ← But no pending read!
```

The command executes successfully, but the result cannot be delivered because the worker never got to call `host_exec_read` (still blocked waiting for `host_exec_start` response).

### Why This Doesn't Affect Other Wasmer Operations

Other Wasmer operations work because:
1. **Compilation**: Happens synchronously before WASM execution
2. **Syscalls**: Most WASI syscalls (file I/O, etc.) are handled within the WASM memory space
3. **spawn_wasm**: Uses dedicated workers that run independently
4. **Async tasks**: Spawned via `wasm_bindgen_futures::spawn_local` which works because the main thread's event loop is free

The `host_exec` case is unique because it requires:
1. Sending a request from a worker
2. **Waiting synchronously** for a response
3. Getting the response back to the same worker

### Potential Solutions

#### 1. SharedArrayBuffer + Atomics (Recommended)

Use `Atomics.wait()` for true synchronous blocking that doesn't prevent message processing:

```rust
// Worker side:
fn host_exec_read_blocking(session: u64) -> i32 {
    let shared_buffer = /* get SharedArrayBuffer for this session */;
    let view = Int32Array::new(&shared_buffer);

    // Atomically wait for the result to be written
    // This blocks the WASM but allows JS event loop to run
    Atomics::wait(&view, 0, 0);  // Wait until value != 0

    // Read the result
    Atomics::load(&view, 1) as i32
}
```

```javascript
// Main thread side:
function sendHostExecResult(sessionId, exitCode) {
    const view = new Int32Array(sharedBuffers.get(sessionId));
    Atomics.store(view, 1, exitCode);  // Store result
    Atomics.store(view, 0, 1);          // Signal ready
    Atomics.notify(view, 0);            // Wake up worker
}
```

**Requirements:**
- COOP/COEP headers in browser (`Cross-Origin-Opener-Policy: same-origin`, `Cross-Origin-Embedder-Policy: require-corp`)
- Works in Node.js without headers
- Requires passing SharedArrayBuffer during worker initialization

#### 2. Asyncify (WASM Transformation)

Use Binaryen's Asyncify to transform WASM to support pause/resume:

```
WASM execution → pause at host_exec → yield to JS event loop →
resume when response arrives → continue execution
```

**Drawbacks:**
- Increases WASM binary size
- Performance overhead for pause/resume
- Complex integration with wasmer runtime

#### 3. Restructure to Polling with setTimeout

Instead of blocking await, use polling with `setTimeout(0)` to yield:

```rust
async fn host_exec_read_polling(session: u64) -> Result<i32, Error> {
    loop {
        // Check if result is available
        if let Some(result) = check_result(session) {
            return Ok(result);
        }

        // Yield to JS event loop via setTimeout(0)
        gloo_timers::future::TimeoutFuture::new(0).await;
    }
}
```

**Drawbacks:**
- `setTimeout(0)` has minimum 4ms delay in browsers (after 5 nested calls)
- Adds significant latency
- Doesn't truly integrate with the async runtime

#### 4. Main Thread Execution (Not Recommended)

Run all WASM on the main thread (no workers):

**Drawbacks:**
- Blocks the UI in browsers
- No parallelism
- Defeats the purpose of the worker-based architecture
